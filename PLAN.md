# План реализации ChatList

Поэтапный порядок действий на основании [PROJECT.md](PROJECT.md).

---

## Этап 0. Подготовка проекта

1. Обновить `requirements.txt`: PyQt6, httpx (или requests), python-dotenv.
2. Создать структуру каталогов:
   - `db.py` — работа с SQLite
   - `models.py` — бизнес-логика моделей и промтов
   - `network.py` — HTTP-запросы к API нейросетей
   - `main.py` — GUI
3. Добавить `.env.example` с именами переменных для API-ключей (без самих ключей).
4. Добавить путь к БД и `.env` в `.gitignore`.

---

## Этап 1. База данных (`db.py`)

1. Реализовать инициализацию SQLite: создание файла БД, применение схемы из [DATABASE.md](DATABASE.md).
2. Написать функции для таблицы **prompts**: добавление, выборка списка, поиск по тексту/тегам.
3. Написать функции для таблицы **models**: CRUD, фильтр активных (`is_active = 1`).
4. Написать функции для таблицы **results**: пакетная вставка выбранных результатов, выборка истории.
5. Написать функции для таблицы **settings**: чтение/запись настроек по ключу.
6. Добавить начальные данные (seed): несколько моделей-заглушек и настройки по умолчанию.

---

## Этап 2. Сетевой слой (`network.py`)

1. Загрузка API-ключей из `.env` по имени переменной из таблицы `models`.
2. Реализовать единый интерфейс отправки промта в модель (функция `send_prompt(model, prompt_text) -> str`).
3. Реализовать адаптеры под разные типы API (минимум OpenAI-совместимый формат; задел под DeepSeek, Groq).
4. Обработка ошибок: таймаут, HTTP-ошибки, пустой ответ — возвращать понятное сообщение вместо падения программы.
5. (Опционально) Логирование запросов: URL, модель, статус, время ответа.

---

## Этап 3. Бизнес-логика (`models.py`)

1. Класс или структура **Prompt**: текст, теги, дата; сохранение в БД.
2. Класс **Model**: name, api_url, api_id, api_key_env, is_active; загрузка активных моделей.
3. Класс **TempResult** (в памяти): model_name, response_text, selected — временная таблица результатов.
4. Функция `run_prompt(prompt_text, active_models) -> list[TempResult]`: параллельная или последовательная отправка во все активные модели.
5. Функция `save_selected_results(prompt_id, temp_results)`: запись строк с `selected = True` в таблицу `results`.
6. Функция `clear_temp_results()`: очистка временной таблицы в памяти.

---

## Этап 4. Главное окно — ввод промта (`main.py`)

1. Заменить заглушку на полноценное окно `MainWindow`.
2. Блок ввода:
   - многострочное поле для нового промта;
   - поле/выбор тегов;
   - выпадающий список сохранённых промтов из БД.
3. Кнопка **Отправить**: при новом промте — сохранить в `prompts` (если нужно), очистить временную таблицу, запустить запросы.
4. Индикатор загрузки на время ожидания ответов от API.

---

## Этап 5. Таблица результатов (временная, в GUI)

1. Виджет `QTableWidget` (или `QTableView` + модель): колонки «Модель», «Ответ», «Выбрать».
2. Заполнение таблицы после получения ответов от `run_prompt`.
3. Чекбокс в каждой строке для поля `selected`.
4. Кнопка **Сохранить**: вызов `save_selected_results`, затем очистка временной таблицы и GUI-таблицы.
5. При вводе нового промта — полная очистка временной таблицы до новых ответов.

---

## Этап 6. Управление моделями (настройки)

1. Отдельное окно или вкладка «Модели»:
   - список моделей из БД;
   - добавление / редактирование / удаление;
   - переключатель `is_active`;
   - поля: name, api_url, api_id, имя переменной окружения для ключа.
2. Валидация: не сохранять модель без name и api_url.

---

## Этап 7. История и настройки

1. Окно или вкладка «История»: таблица сохранённых результатов из `results` с привязкой к промту и модели.
2. Окно «Настройки»: редактирование записей из таблицы `settings` (таймаут запроса, путь к БД и т.д.).
3. Поиск и сортировка в таблицах истории, промтов и моделей (опционально, см. PROJECT.md §6).

---

## Этап 8. Дополнительные функции (опционально)

1. Экспорт выбранных результатов в Markdown и JSON.
2. Расширение адаптеров API для DeepSeek, Groq и других провайдеров.
3. Файл логов запросов с ротацией.

---

## Этап 9. Тестирование и финализация

1. Проверить сценарий: ввод промта → отправка → выбор строк → сохранение → новый запрос.
2. Проверить работу без API-ключей (корректные сообщения об ошибках).
3. Проверить выбор сохранённого промта из списка.
4. Обновить `README.md`: установка, настройка `.env`, запуск.

---

## Порядок файлов при разработке

```
db.py          →  models.py  →  network.py  →  main.py (по частям)
     ↓               ↓              ↓
DATABASE.md    временные       адаптеры API
               структуры
```

Рекомендуется идти строго по этапам 1 → 2 → 3, затем собирать GUI (этапы 4–7), чтобы каждый слой можно было проверить отдельно до подключения к интерфейсу.

## Этап 10. AI-ассистент для улучшения промтов

По спецификации [PROJECT17.md](PROJECT17.md) §7. Функция помогает пользователю улучшить текст запроса перед отправкой в несколько моделей.

### Цель

Пользователь вводит исходный промт, нажимает **«Улучшить промт»**, получает от одной выбранной модели:
- улучшенную версию;
- 2–3 альтернативные переформулировки;
- при необходимости — варианты под разные задачи (код, анализ, креатив).

Выбранный вариант подставляется в поле ввода одним кликом.

### Архитектура (вписывается в текущий проект)

```
PromptTab (main.py)
    ↓ кнопка «Улучшить промт»
ImprovePromptWorker (QThread)
    ↓
prompt_assistant.py (новый модуль) или models.py
    ↓
network.send_prompt() — существующий OpenRouter-клиент
    ↓
ImprovePromptDialog — окно с вариантами
```

Новые изменения не требуют отдельной таблицы в БД: достаточно настройки «модель для ассистента» в `settings`.

---

### Подэтап 10.1. Настройки (`db.py`, `tabs.py`)

1. Добавить ключ настройки `improve_prompt_model_id` в `DEFAULT_SETTINGS` (`db.py`):
   - значение по умолчанию: `""` (пусто) или ID первой активной OpenRouter-модели.
2. В `SettingsTab` (`tabs.py`) добавить выпадающий список **«Модель для улучшения промтов»**:
   - список активных моделей из `db.list_models(active_only=True)`;
   - сохранение выбранного `model_id` через `db.set_setting("improve_prompt_model_id", ...)`.
3. Валидация: если модель не выбрана или неактивна — показывать предупреждение при нажатии «Улучшить промт».

---

### Подэтап 10.2. Бизнес-логика (`prompt_assistant.py` или `models.py`)

1. Создать модуль `prompt_assistant.py` (предпочтительно, чтобы не раздувать `models.py`):
   - датакласс `PromptImprovementResult`:
     - `original: str`
     - `improved: str`
     - `alternatives: list[str]` (2–3 элемента)
     - `adaptations: dict[str, str]` — ключи: `code`, `analysis`, `creative` (опционально).
2. Функция `build_improvement_prompt(user_prompt: str) -> str`:
   - системная инструкция для модели: улучшить промт, вернуть структурированный JSON;
   - требование к формату ответа (см. ниже).
3. Функция `improve_prompt(user_prompt: str, model: Model) -> PromptImprovementResult`:
   - загрузить модель по `improve_prompt_model_id` из настроек;
   - вызвать `network.send_prompt(model.to_dict(), meta_prompt)`;
   - распарсить ответ.
4. Функция `parse_improvement_response(raw: str, original: str) -> PromptImprovementResult`:
   - попытка `json.loads()` из ответа модели;
   - fallback: если JSON не распознан — весь текст считать `improved`, `alternatives` пустой;
   - обрезка markdown-обёрток (`` ```json ... ``` ``).

**Ожидаемый JSON от модели:**

```json
{
  "improved": "улучшенный промт",
  "alternatives": ["вариант 1", "вариант 2", "вариант 3"],
  "adaptations": {
    "code": "промт для задач программирования",
    "analysis": "промт для аналитики",
    "creative": "промт для креативных задач"
  }
}
```

5. Проверки до отправки:
   - исходный промт не пустой;
   - длина промта в разумных пределах (например, до 8000 символов);
   - у выбранной модели есть API-ключ в `.env`.

---

### Подэтап 10.3. Сетевой слой (`network.py`)

1. **Не создавать отдельный HTTP-клиент** — переиспользовать `network.send_prompt()`.
2. Убедиться, что meta-промт передаётся как обычный `prompt_text` (ассистент — это тот же chat/completions-запрос).
3. Ошибки (таймаут, HTTP, пустой ответ) пробрасывать наверх и показывать в UI так же, как при обычной отправке.

---

### Подэтап 10.4. Интерфейс (`main.py`, отдельный диалог)

#### 10.4.1. Кнопка на вкладке «Запрос»

1. В `PromptTab` (`main.py`) рядом с полем «Новый промт» добавить кнопку **«Улучшить промт»**.
2. Разместить в `buttons_row` или отдельной строке над полем ввода (логично — сразу под полем промта, до «Отправить»).
3. При нажатии:
   - взять текст из `self.prompt_edit`;
   - если пусто — `QMessageBox.warning`;
   - запустить фоновый поток `ImprovePromptWorker`.

#### 10.4.2. Фоновый поток

1. Класс `ImprovePromptWorker(QThread)` по аналогии с `SendWorker`:
   - `finished = pyqtSignal(PromptImprovementResult)`
   - `failed = pyqtSignal(str)`
2. На время работы:
   - показать `QProgressBar` (можно переиспользовать существующий);
   - заблокировать кнопки «Улучшить промт» и «Отправить».

#### 10.4.3. Диалог результатов (`ImprovePromptDialog`)

Создать в `prompt_assistant.py` или отдельном `prompt_assistant_ui.py`:

1. Заголовок: **«Улучшение промта»**.
2. Блоки (только чтение, многострочные `QTextEdit` или `QLabel` с переносом):
   - **Исходный промт**;
   - **Улучшенный** + кнопка **«Подставить»**;
   - **Альтернативы** (список из 2–3 вариантов), у каждого — **«Подставить»**;
   - **Адаптации** (вкладки или секции: Код / Анализ / Креатив), у каждой — **«Подставить»**.
3. Кнопка **«Закрыть»** без сохранения.
4. Сигнал `variant_selected(str)` → в `PromptTab` подставить текст в `prompt_edit` и закрыть диалог.

---

### Подэтап 10.5. Связка с остальным приложением

1. Подстановка варианта **не сохраняет** промт в БД автоматически — только заменяет текст в поле ввода (пользователь сам решает отправлять или сохранять).
2. После подстановки можно сразу нажать **«Отправить»** — стандартный сценарий.
3. Опционально: при подстановке улучшенного варианта обновить комбобокс «— новый промт —» (сбросить выбранный сохранённый промт).

---

### Подэтап 10.6. Meta-промт (шаблон)

Зафиксировать шаблон в `prompt_assistant.py` как константу `IMPROVE_PROMPT_TEMPLATE`:

- роль: эксперт по prompt engineering;
- задачи: ясность, конкретность, структура, контекст;
- вернуть **только JSON** без пояснений;
- язык ответа — тот же, что у исходного промта (или русский, если промт на русском).

Пример структуры инструкции (черновик для реализации):

```
Ты — ассистент по улучшению промтов. Пользователь прислал промт:
---
{user_prompt}
---
Верни JSON с полями: improved, alternatives (массив из 2-3 строк),
adaptations (объект с ключами code, analysis, creative).
Ответ — только валидный JSON, без markdown.
```

---

### Подэтап 10.7. Тестирование

1. Пустой промт → предупреждение, запрос не уходит.
2. Модель не выбрана в настройках → понятное сообщение.
3. Нет API-ключа → сообщение об ошибке из `network.send_prompt`.
4. Корректный JSON в ответе → все блоки заполнены в диалоге.
5. Некорректный JSON → fallback: весь ответ в поле «Улучшенный».
6. «Подставить» → текст появляется в поле ввода.
7. Повторное улучшение после подстановки работает.
8. Параллельно с «Отправить» не допускать двойной запуск (блокировка UI).

---

### Порядок реализации (кратко)

```
1. db.py + SettingsTab     → настройка improve_prompt_model_id
2. prompt_assistant.py     → шаблон, парсинг, improve_prompt()
3. ImprovePromptWorker     → фоновый вызов
4. ImprovePromptDialog     → UI результатов
5. PromptTab               → кнопка «Улучшить промт» и связка
6. Ручное тестирование     → сценарии из 10.7
```

### Затрагиваемые файлы

| Файл | Изменения |
|------|-----------|
| `db.py` | ключ `improve_prompt_model_id` в `DEFAULT_SETTINGS` |
| `prompt_assistant.py` | **новый** — логика ассистента |
| `network.py` | без изменений (переиспользование) |
| `tabs.py` | выбор модели ассистента в настройках |
| `main.py` | кнопка, worker, открытие диалога |
| `prompt_assistant_ui.py` | **опционально** — диалог результатов |

### Зависимости

- Этапы 1–4 должны быть готовы (БД, сеть, `PromptTab`, активные модели).
- Для работы ассистента нужна хотя бы одна активная модель OpenRouter с ключом в `.env`.
