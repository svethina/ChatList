# План реализации ChatList

Поэтапный порядок действий на основании [PROJECT.md](PROJECT.md).

---

## Этап 0. Подготовка проекта

1. Обновить `requirements.txt`: PyQt6, httpx (или requests), python-dotenv.
2. Создать структуру каталогов:
   - `db.py` — работа с SQLite
   - `models.py` — бизнес-логика моделей и промтов
   - `network.py` — HTTP-запросы к API нейросетей
   - `main.py` — GUI
3. Добавить `.env.example` с именами переменных для API-ключей (без самих ключей).
4. Добавить путь к БД и `.env` в `.gitignore`.

---

## Этап 1. База данных (`db.py`)

1. Реализовать инициализацию SQLite: создание файла БД, применение схемы из [DATABASE.md](DATABASE.md).
2. Написать функции для таблицы **prompts**: добавление, выборка списка, поиск по тексту/тегам.
3. Написать функции для таблицы **models**: CRUD, фильтр активных (`is_active = 1`).
4. Написать функции для таблицы **results**: пакетная вставка выбранных результатов, выборка истории.
5. Написать функции для таблицы **settings**: чтение/запись настроек по ключу.
6. Добавить начальные данные (seed): несколько моделей-заглушек и настройки по умолчанию.

---

## Этап 2. Сетевой слой (`network.py`)

1. Загрузка API-ключей из `.env` по имени переменной из таблицы `models`.
2. Реализовать единый интерфейс отправки промта в модель (функция `send_prompt(model, prompt_text) -> str`).
3. Реализовать адаптеры под разные типы API (минимум OpenAI-совместимый формат; задел под DeepSeek, Groq).
4. Обработка ошибок: таймаут, HTTP-ошибки, пустой ответ — возвращать понятное сообщение вместо падения программы.
5. (Опционально) Логирование запросов: URL, модель, статус, время ответа.

---

## Этап 3. Бизнес-логика (`models.py`)

1. Класс или структура **Prompt**: текст, теги, дата; сохранение в БД.
2. Класс **Model**: name, api_url, api_id, api_key_env, is_active; загрузка активных моделей.
3. Класс **TempResult** (в памяти): model_name, response_text, selected — временная таблица результатов.
4. Функция `run_prompt(prompt_text, active_models) -> list[TempResult]`: параллельная или последовательная отправка во все активные модели.
5. Функция `save_selected_results(prompt_id, temp_results)`: запись строк с `selected = True` в таблицу `results`.
6. Функция `clear_temp_results()`: очистка временной таблицы в памяти.

---

## Этап 4. Главное окно — ввод промта (`main.py`)

1. Заменить заглушку на полноценное окно `MainWindow`.
2. Блок ввода:
   - многострочное поле для нового промта;
   - поле/выбор тегов;
   - выпадающий список сохранённых промтов из БД.
3. Кнопка **Отправить**: при новом промте — сохранить в `prompts` (если нужно), очистить временную таблицу, запустить запросы.
4. Индикатор загрузки на время ожидания ответов от API.

---

## Этап 5. Таблица результатов (временная, в GUI)

1. Виджет `QTableWidget` (или `QTableView` + модель): колонки «Модель», «Ответ», «Выбрать».
2. Заполнение таблицы после получения ответов от `run_prompt`.
3. Чекбокс в каждой строке для поля `selected`.
4. Кнопка **Сохранить**: вызов `save_selected_results`, затем очистка временной таблицы и GUI-таблицы.
5. При вводе нового промта — полная очистка временной таблицы до новых ответов.

---

## Этап 6. Управление моделями (настройки)

1. Отдельное окно или вкладка «Модели»:
   - список моделей из БД;
   - добавление / редактирование / удаление;
   - переключатель `is_active`;
   - поля: name, api_url, api_id, имя переменной окружения для ключа.
2. Валидация: не сохранять модель без name и api_url.

---

## Этап 7. История и настройки

1. Окно или вкладка «История»: таблица сохранённых результатов из `results` с привязкой к промту и модели.
2. Окно «Настройки»: редактирование записей из таблицы `settings` (таймаут запроса, путь к БД и т.д.).
3. Поиск и сортировка в таблицах истории, промтов и моделей (опционально, см. PROJECT.md §6).

---

## Этап 8. Дополнительные функции (опционально)

1. Экспорт выбранных результатов в Markdown и JSON.
2. Расширение адаптеров API для DeepSeek, Groq и других провайдеров.
3. Файл логов запросов с ротацией.

---

## Этап 9. Тестирование и финализация

1. Проверить сценарий: ввод промта → отправка → выбор строк → сохранение → новый запрос.
2. Проверить работу без API-ключей (корректные сообщения об ошибках).
3. Проверить выбор сохранённого промта из списка.
4. Обновить `README.md`: установка, настройка `.env`, запуск.

---

## Порядок файлов при разработке

```
db.py          →  models.py  →  network.py  →  main.py (по частям)
     ↓               ↓              ↓
DATABASE.md    временные       адаптеры API
               структуры
```

Рекомендуется идти строго по этапам 1 → 2 → 3, затем собирать GUI (этапы 4–7), чтобы каждый слой можно было проверить отдельно до подключения к интерфейсу.
